üèóÔ∏è SQLite Architecture

What is SQLite?
SQLite is a serverless, self-contained SQL database engine. Unlike PostgreSQL/MySQL, it doesn't require a separate server process. The entire database is stored in a single file on disk, making it perfect for embedded applications, mobile apps, and desktop software.
Serverless Architecture
No Server Process: Reads/writes directly to disk
Zero Configuration: No setup or administration
Single File: Entire DB in one .db/.sqlite file
Cross-platform: Works on all major OS

SQLite vs PostgreSQL/MySQL

Feature SQLite PostgreSQL/MySQL
Architecture Serverless, embedded Client-server model
Configuration Zero-config Requires setup & tuning
Concurrency Limited (file-level locking) High (row-level locking)
Data Types Dynamic typing (5 storage classes) Static typing (strict types)
Size Limit 281 TB (practical: ~1 TB) Virtually unlimited
Best For Embedded, mobile, desktop apps Web apps, enterprise systems
üí° Key Point: SQLite is not a replacement for client-server databases. It's designed for different use cases: local storage, embedded systems, and applications that need a lightweight, self-contained database.

üîí ACID Properties in SQLite

Property Description SQLite Implementation
Atomicity All or nothing transactions Rollback journal or WAL mode ensures atomic commits
Consistency Database remains valid after transactions Constraints (NOT NULL, UNIQUE, CHECK, FK) enforced
Isolation Concurrent transactions don't interfere Serializable isolation (file-level locking)
Durability Committed data persists after crashes fsync() calls ensure data written to disk
Transaction Modes in SQLite:
‚Ä¢ BEGIN DEFERRED - Default, locks acquired on first read/write
‚Ä¢ BEGIN IMMEDIATE - Acquires RESERVED lock immediately
‚Ä¢ BEGIN EXCLUSIVE - Acquires EXCLUSIVE lock immediately

üéØ Data Types & Type Affinity

Dynamic Typing System
Unlike PostgreSQL/MySQL, SQLite uses dynamic typing. Values are stored in one of 5 storage classes, and columns have "type affinity" (preferred type) rather than strict types.

5 Storage Classes

Storage Class Description Examples
NULL Null value NULL
INTEGER Signed integer (1, 2, 3, 4, 6, or 8 bytes) -9223372036854775808 to 9223372036854775807
REAL Floating point (8-byte IEEE) 3.14159, -273.15
TEXT Text string (UTF-8, UTF-16BE, UTF-16LE) 'Hello', "SQLite"
BLOB Binary Large Object Images, files, binary data

Type Affinity Rules

-- INTEGER affinity CREATE TABLE example1 ( id INTEGER, count INT, num BIGINT ); -- TEXT affinity CREATE TABLE example2 ( name TEXT, description VARCHAR(255), notes CHAR(100) ); -- REAL affinity CREATE TABLE example3 ( price REAL, rating DOUBLE, score FLOAT ); -- BLOB affinity CREATE TABLE example4 ( data BLOB ); -- NUMERIC affinity (INTEGER if possible, else REAL) CREATE TABLE example5 ( value NUMERIC, amount DECIMAL(10,2) );
‚ö†Ô∏è Important: SQLite ignores length specifications like VARCHAR(255). All TEXT values can be any length. The type name only affects affinity, not storage.

üõ†Ô∏è DDL Operations

CREATE TABLE

-- Basic table creation CREATE TABLE users ( id INTEGER PRIMARY KEY AUTOINCREMENT, username TEXT NOT NULL UNIQUE, email TEXT NOT NULL, age INTEGER, created_at TEXT DEFAULT CURRENT_TIMESTAMP ); -- Table with constraints CREATE TABLE products ( id INTEGER PRIMARY KEY, name TEXT NOT NULL, price REAL CHECK(price > 0), category TEXT DEFAULT 'general', stock INTEGER DEFAULT 0 ); -- Table with foreign key CREATE TABLE orders ( id INTEGER PRIMARY KEY, user_id INTEGER, product_id INTEGER, quantity INTEGER, FOREIGN KEY (user_id) REFERENCES users(id), FOREIGN KEY (product_id) REFERENCES products(id) );
üí° AUTOINCREMENT vs PRIMARY KEY:
‚Ä¢ INTEGER PRIMARY KEY - Reuses deleted IDs, faster
‚Ä¢ INTEGER PRIMARY KEY AUTOINCREMENT - Never reuses IDs, slightly slower

ALTER TABLE

-- Add column (SQLite 3.2.0+) ALTER TABLE users ADD COLUMN phone TEXT; -- Rename table ALTER TABLE users RENAME TO customers; -- Rename column (SQLite 3.25.0+) ALTER TABLE customers RENAME COLUMN username TO user_name; -- Drop column (SQLite 3.35.0+) ALTER TABLE customers DROP COLUMN phone;
‚ö†Ô∏è Limitation: SQLite has limited ALTER TABLE support compared to PostgreSQL. You cannot modify column types or add constraints to existing columns directly. Use the "recreate table" pattern for complex changes.

DROP TABLE

-- Drop table DROP TABLE orders; -- Drop if exists (safe) DROP TABLE IF EXISTS orders;

üìù CRUD Operations

INSERT

-- Single row insert INSERT INTO users (username, email, age) VALUES ('john_doe', 'john@example.com', 25); -- Multiple rows insert INSERT INTO users (username, email, age) VALUES ('alice', 'alice@example.com', 30), ('bob', 'bob@example.com', 28), ('charlie', 'charlie@example.com', 35); -- Insert or ignore (skip if duplicate) INSERT OR IGNORE INTO users (username, email) VALUES ('john_doe', 'john@example.com'); -- Insert or replace (upsert) INSERT OR REPLACE INTO users (id, username, email) VALUES (1, 'john_updated', 'john@example.com');

SELECT

-- Basic select SELECT * FROM users; -- Select specific columns SELECT username, email FROM users; -- With WHERE clause SELECT * FROM users WHERE age > 25; -- With ORDER BY SELECT * FROM users ORDER BY age DESC; -- With LIMIT SELECT * FROM users LIMIT 10 OFFSET 5;

UPDATE

-- Update single column UPDATE users SET age = 26 WHERE username = 'john_doe'; -- Update multiple columns UPDATE users SET email = 'newemail@example.com', age = 27 WHERE id = 1; -- Update all rows (be careful!) UPDATE users SET age = age + 1;

DELETE

-- Delete specific rows DELETE FROM users WHERE age < 18; -- Delete all rows (be careful!) DELETE FROM users; -- Delete with subquery DELETE FROM orders WHERE user_id IN (SELECT id FROM users WHERE age < 18);

üíª SQLite CLI Commands

Command Description Example
.help Show all available commands .help
.open Open database file .open mydb.db
.tables List all tables .tables
.schema Show table schema .schema users

.quit Exit SQLite CLI .quit
