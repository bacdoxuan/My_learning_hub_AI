üîë Shard Key - The Most Important Decision
Shard Key determines how data is distributed across shards. CANNOT be changed after sharding!
Shard Key Properties
Property Description Why Important
Cardinality Number of unique values High cardinality = better distribution
Frequency How often values appear Even frequency = balanced shards
Monotonicity Values increase/decrease over time Avoid monotonic keys (hotspots)
Good vs Bad Shard Keys
‚úÖ Good Shard Keys
{userId: 1} - High cardinality
{email: 1} - Unique, distributed
{country: 1, userId: 1} - Compound
{customerId: "hashed"} - Hashed
‚ùå Bad Shard Keys
{_id: 1} - Monotonically increasing
{createdAt: 1} - Timestamp (monotonic)
{status: 1} - Low cardinality
{country: 1} - Uneven distribution
Shard Key Strategies
Ranged Sharding Hashed Sharding Compound Shard Key Zone Sharding

üéØ Targeted vs Broadcast Queries
Targeted Query (Best)
// Query includes shard key - goes to 1 shard
db.users.find({ userId: 12345 })

// Shard key: {userId: "hashed"}
// ‚úÖ Routed to single shard
Broadcast Query (Scatter-Gather)
// Query without shard key - goes to ALL shards
db.users.find({ email: "user@example.com" })

// ‚ö†Ô∏è Queries all shards, merges results
// Slower, more resource intensive
Optimize Queries
Always include shard key in queries when possible
Create indexes on non-shard-key fields
Use compound shard keys for common query patterns
Avoid $where and JavaScript expressions
‚úÖ Use explain() to check if query is targeted or broadcast
‚ö†Ô∏è Sharding Limitations
Cannot Change
Shard key after sharding (immutable)
Shard key values (immutable in MongoDB < 4.2)
Unique Indexes
// Unique index must include shard key
db.users.createIndex(
  { userId: 1, email: 1 },
  { unique: true }
)

// Shard key: {userId: 1}
// ‚úÖ Valid - includes shard key
Transactions
Multi-document transactions supported (4.2+)
Distributed transactions slower than single-shard
Consider transaction scope when designing shard key
Aggregation
$lookup across shards possible but expensive
$merge, $out require shard key
Some stages must run on primary shard
‚ö†Ô∏è Plan shard key carefully - it affects query performance and data distribution
‚≠ê Sharding Best Practices
‚úÖ Planning
Shard only when necessary (vertical scaling first)
Choose shard key based on query patterns
Test shard key with production-like data
Plan for 2-3x growth
Consider zone sharding for compliance
Document shard key rationale
‚úÖ Shard Key Selection
High cardinality (many unique values)
Even frequency distribution
Avoid monotonically increasing keys
Include in most queries
Consider compound keys
Use hashed for write-heavy workloads
‚úÖ Operations
Monitor chunk distribution regularly
Set balancing windows for off-peak
Backup from secondary members
Test failover scenarios
Monitor query patterns (targeted vs broadcast)
Scale shards independently
Common Shard Key Patterns
Pattern Shard Key Use Case
User-based {userId: "hashed"} User data, sessions, profiles
Geographic {country: 1, userId: 1} Multi-region apps, compliance
Time-series {deviceId: 1, timestamp: 1} IoT data, logs, metrics
Tenant-based {tenantId: 1, ...} Multi-tenant SaaS
Category-based {category: 1, productId: 1} E-commerce, content platforms
