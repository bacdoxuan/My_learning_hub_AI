üîÑ Auto Statistics & Updates

Automatic Statistics Management

Auto Create Statistics
Automatic creation: SQL Server creates statistics when needed for query optimization
Threshold-based: Triggered when table has data and queries reference columns
Naming convention: _WA_Sys_ followed by column ID and table ID
Database setting: AUTO_CREATE_STATISTICS option (default ON)
Scope: Applies to all user databases
Auto Update Statistics
Automatic updates: Statistics refresh when data changes significantly
Threshold trigger: 20% of rows changed + 500 rows minimum
Database setting: AUTO_UPDATE_STATISTICS option (default ON)
Asynchronous option: AUTO_UPDATE_STATISTICS_ASYNC for non-blocking updates
Performance impact: Updates can cause query recompilation
-- Check auto statistics settings SELECT name, is_auto_create_stats_on, is_auto_update_stats_on, is_auto_update_stats_async_on FROM sys.databases WHERE name = 'YourDatabase'; -- Enable/disable auto statistics ALTER DATABASE YourDatabase SET AUTO_CREATE_STATISTICS ON; ALTER DATABASE YourDatabase SET AUTO_UPDATE_STATISTICS ON; ALTER DATABASE YourDatabase SET AUTO_UPDATE_STATISTICS_ASYNC ON; -- View existing statistics SELECT object_name(s.object_id) AS TableName, s.name AS StatisticsName, s.auto_created, s.user_created, STATS_DATE(s.object_id, s.stats_id) AS LastUpdated FROM sys.stats s WHERE object_name(s.object_id) = 'YourTable';

üìä Histogram Insights

Data Distribution Analysis

Histogram Structure & Usage
Steps and ranges: Up to 200 steps showing data distribution
RANGE_HI_KEY: Upper bound of each histogram step
RANGE_ROWS: Number of rows within the step range
EQ_ROWS: Number of rows equal to RANGE_HI_KEY
DISTINCT_RANGE_ROWS: Distinct values within the step
AVG_RANGE_ROWS: Average rows per distinct value in range
Cardinality estimation: Used by optimizer for row count predictions
-- View histogram details DBCC SHOW_STATISTICS ('Sales.Orders', 'IX_Orders_OrderDate'); -- Or using system function SELECT hist_step_id, range_hi_key, range_rows, eq_rows, distinct_range_rows, avg_range_rows / avg_range_rows AS avg_rows_per_value FROM sys.dm_db_stats_histogram(OBJECT_ID('Sales.Orders'), STATS_ID(OBJECT_ID('Sales.Orders'), 'IX_Orders_OrderDate')); -- Density information DBCC SHOW_STATISTICS ('Sales.Orders', 'IX_Orders_OrderDate') WITH DENSITY_VECTOR; -- Understanding histogram: -- RANGE_HI_KEY: 2023-01-15 has EQ_ROWS = 5 (exact matches) -- Values 2023-01-10 to 2023-01-15 have RANGE_ROWS = 20 (within range) -- DISTINCT_RANGE_ROWS = 6 (6 different dates in that range)

üîÑ When to Update Statistics

Manual Statistics Maintenance

Update Triggers & Best Practices
Data changes: After significant inserts, updates, deletes (>20% + 500 rows)
Schema changes: After adding/dropping indexes or columns
Query performance: When execution plans show inaccurate estimates
Bulk operations: After BULK INSERT, TRUNCATE, or large data loads
Filtered statistics: When data distribution changes in filtered subsets
Maintenance windows: Schedule updates during low-activity periods
-- Update specific statistics UPDATE STATISTICS Sales.Orders IX_Orders_OrderDate; -- Update all statistics on a table UPDATE STATISTICS Sales.Orders; -- Fullscan for complete accuracy (slower) UPDATE STATISTICS Sales.Orders IX_Orders_OrderDate WITH FULLSCAN; -- Sample-based update (faster, less accurate) UPDATE STATISTICS Sales.Orders IX_Orders_OrderDate WITH SAMPLE 50 PERCENT; -- Update all statistics in database EXEC sp_updatestats; -- Check statistics modification counter SELECT object_name(object_id) AS TableName, stats_id, modification_counter, rows, modification_counter * 100.0 / rows AS PercentModified FROM sys.dm_db_stats_properties(OBJECT_ID('Sales.Orders'), NULL) WHERE modification_counter > 0;

üìà Sampling Methods

Statistics Collection Strategies

Full Scan Sampling
Complete accuracy: Scans all rows for perfect statistics
Performance cost: High I/O and time for large tables
Best for: Critical tables, small datasets, accurate histograms
Default behavior: Used when table size allows
Resource intensive: May impact production performance
Sample-Based Collection
Performance tradeoff: Faster collection with reduced accuracy
Sampling rates: Default algorithm chooses appropriate sample size
Manual control: Specify PERCENT or ROWS for custom sampling
Best for: Large tables, maintenance windows, acceptable inaccuracy
Quality considerations: Smaller samples may miss data distribution nuances
-- Default sampling (SQL Server chooses) UPDATE STATISTICS LargeTable; -- Explicit percentage sampling UPDATE STATISTICS LargeTable WITH SAMPLE 30 PERCENT; -- Fixed number of rows UPDATE STATISTICS LargeTable WITH SAMPLE 10000 ROWS; -- Full scan for maximum accuracy UPDATE STATISTICS LargeTable WITH FULLSCAN; -- Check sampling rate used SELECT sp.last_updated, sp.rows, sp.rows_sampled, CASE WHEN sp.rows_sampled = sp.rows THEN 100.0 ELSE sp.rows_sampled * 100.0 / sp.rows END AS SamplePercent FROM sys.dm_db_stats_properties(OBJECT_ID('LargeTable'), NULL) sp;

‚ö° Impacts of Poor Cardinality Estimation

Performance Consequences

Estimation Errors & Their Effects
Suboptimal plans: Wrong join algorithms, incorrect operator choices
Memory issues: Underestimated row counts cause spills to tempdb
Resource waste: Overestimated plans allocate excessive memory
Query timeouts: Plans that should be fast become slow and vice versa
Blocking/deadlocks: Longer-running queries increase concurrency issues
Index selection: Wrong indexes chosen for data access
-- Detect cardinality estimation issues SELECT qs.execution_count, qs.total_logical_reads, qs.total_logical_writes, qs.total_elapsed_time / qs.execution_count AS avg_elapsed_time, SUBSTRING(st.text, (qs.statement_start_offset/2)+1, ((CASE qs.statement_end_offset WHEN -1 THEN DATALENGTH(st.text) ELSE qs.statement_end_offset END - qs.statement_start_offset)/2) + 1) AS query_text FROM sys.dm_exec_query_stats qs CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st WHERE qs.last_execution_time > DATEADD(hour, -24, GETDATE()) ORDER BY qs.total_logical_reads DESC; -- Check for spills in execution plans -- Look for "Hash Warning" or "Sort Warning" in actual plans -- Query with cardinality mismatch SELECT * FROM LargeTable WHERE FilteredColumn = 'RareValue'; -- Statistics may underestimate rows
üí° Statistics Best Practices: Keep auto-update enabled but supplement with manual updates for critical tables. Use FULLSCAN for important statistics. Monitor for cardinality estimation errors through execution plan analysis.

Scenario Recommended Approach Rationale
OLTP Database Auto-update with async option Minimize blocking during updates
Data Warehouse Scheduled FULLSCAN updates Maximize accuracy for complex queries
Large Tables Sample-based updates (30-50%) Balance accuracy with maintenance time
Critical Queries Manual updates after data loads Ensure optimal plans for important workloads
