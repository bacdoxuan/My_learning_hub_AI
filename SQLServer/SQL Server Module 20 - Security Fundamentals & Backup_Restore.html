üîê Security Fundamentals

Authentication & Authorization

Logins & Users
Logins: Server-level authentication (SQL Server or Windows)
Users: Database-level principals mapped to logins
Contained users: Users without server logins for portability
Guest user: Default user for unauthenticated access
Orphaned users: Users without corresponding logins
Best practice: Use Windows authentication when possible
Roles & Permissions
Fixed server roles: sysadmin, serveradmin, etc.
Fixed database roles: db_owner, db_datareader, etc.
User-defined roles: Custom roles for specific permissions
Permissions: GRANT, DENY, REVOKE on objects
Schema permissions: Control access to schema objects
Execution permissions: Required for stored procedures

Least Privilege & Ownership Chaining

Security Principles
Least privilege: Grant minimum permissions required for tasks
Defense in depth: Multiple security layers
Ownership chaining: Object owner permissions flow through chains
Cross-database chaining: Requires TRUSTWORTHY or certificates
Impersonation: EXECUTE AS for temporary permission elevation
Auditing: Track permission usage and changes
-- Create SQL Server login CREATE LOGIN AppUser WITH PASSWORD = 'StrongPassword123!'; -- Create Windows login CREATE LOGIN [DOMAIN\User] FROM WINDOWS; -- Create database user USE MyDatabase; CREATE USER AppUser FOR LOGIN AppUser; -- Create user-defined database role CREATE ROLE SalesRole; ALTER ROLE SalesRole ADD MEMBER AppUser; -- Grant permissions using least privilege GRANT SELECT ON Sales.Orders TO SalesRole; GRANT EXECUTE ON Sales.GetOrderDetails TO SalesRole; -- Deny sensitive permissions DENY SELECT ON Sales.SalaryInfo TO SalesRole; -- Ownership chaining example CREATE SCHEMA Sales AUTHORIZATION dbo; CREATE TABLE Sales.Orders (...); CREATE PROCEDURE Sales.GetOrders AS BEGIN SELECT * FROM Sales.Orders; -- Inherits dbo permissions END; -- Grant execute on procedure only GRANT EXECUTE ON Sales.GetOrders TO SalesRole; -- EXECUTE AS for impersonation CREATE PROCEDURE Sales.AdminTask WITH EXECUTE AS OWNER AS BEGIN -- Runs with procedure owner's permissions UPDATE Sales.Orders SET Status = 'Approved'; END; -- Check effective permissions SELECT * FROM fn_my_permissions('Sales.Orders', 'OBJECT'); -- Audit login events SELECT login_name, login_time, session_id FROM sys.dm_exec_sessions WHERE login_name IS NOT NULL ORDER BY login_time DESC;

üíæ Backup & Restore Operations

Backup Types & Strategies

Backup Types
Full backup: Complete database backup, foundation for differentials
Differential backup: Only changes since last full backup
Transaction log backup: Captures transaction log changes
File/filegroup backup: Backup specific files or filegroups
Partial backup: Primary filegroup + read-write secondary filegroups
Copy-only backup: Doesn't affect backup sequence

Restore Sequences & Verification

Restore Process
Point-in-time recovery: Restore to specific transaction
Recovery models: Simple, Full, Bulk-logged affect restore options
Tail-log backup: Capture active transactions before restore
Verification: RESTORE VERIFYONLY and CHECKSUM
NORECOVERY/RECOVERY: Control transaction rollback
Testing restores: Regularly test restore procedures
-- Full database backup BACKUP DATABASE MyDatabase TO DISK = 'D:\Backups\MyDatabase_Full.bak' WITH NAME = 'Full Backup', DESCRIPTION = 'Complete database backup', STATS = 10; -- Differential backup BACKUP DATABASE MyDatabase TO DISK = 'D:\Backups\MyDatabase_Diff.bak' WITH DIFFERENTIAL, NAME = 'Differential Backup'; -- Transaction log backup BACKUP LOG MyDatabase TO DISK = 'D:\Backups\MyDatabase_Log.trn' WITH NAME = 'Log Backup'; -- Copy-only backup (doesn't break chain) BACKUP DATABASE MyDatabase TO DISK = 'D:\Backups\MyDatabase_Copy.bak' WITH COPY_ONLY; -- Restore sequence example -- 1. Restore full backup with NORECOVERY RESTORE DATABASE MyDatabase FROM DISK = 'D:\Backups\MyDatabase_Full.bak' WITH NORECOVERY, REPLACE; -- 2. Restore differential if available RESTORE DATABASE MyDatabase FROM DISK = 'D:\Backups\MyDatabase_Diff.bak' WITH NORECOVERY; -- 3. Restore transaction logs in sequence RESTORE LOG MyDatabase FROM DISK = 'D:\Backups\MyDatabase_Log1.trn' WITH NORECOVERY; RESTORE LOG MyDatabase FROM DISK = 'D:\Backups\MyDatabase_Log2.trn' WITH NORECOVERY; -- 4. Final restore with RECOVERY RESTORE DATABASE MyDatabase WITH RECOVERY; -- Point-in-time restore RESTORE LOG MyDatabase FROM DISK = 'D:\Backups\MyDatabase_Log.trn' WITH RECOVERY, STOPAT = '2024-01-15 10:00:00'; -- Verify backup integrity RESTORE VERIFYONLY FROM DISK = 'D:\Backups\MyDatabase_Full.bak'; -- Backup with checksum BACKUP DATABASE MyDatabase TO DISK = 'D:\Backups\MyDatabase_Checksum.bak' WITH CHECKSUM; -- Monitor backup status SELECT database_name, backup_start_date, backup_finish_date, type, CAST(backup_size / 1024 / 1024 AS DECIMAL(10,2)) AS BackupMB FROM msdb.dbo.backupset WHERE database_name = 'MyDatabase' ORDER BY backup_start_date DESC;

‚úÖ Security & Backup Checklist

  • Implement Windows authentication for better security
  • Create appropriate server and database roles
  • Apply principle of least privilege to all users
  • Use ownership chaining for stored procedure security
  • Audit login and permission changes
  • Choose appropriate recovery model for each database
  • Implement regular full backup schedules
  • Configure differential backups between full backups
  • Set up transaction log backups for Full recovery
  • Test backup integrity with RESTORE VERIFYONLY
  • Use CHECKSUM for backup verification
  • Document restore sequences and procedures
  • Test complete restore scenarios regularly
  • Monitor backup success and duration
  • Implement backup compression to save space
  • Use encryption for sensitive backup files
  • Store backups on separate media from data files
  • Implement backup retention policies
  • Monitor for orphaned users after restores
  • Document security policies and procedures
