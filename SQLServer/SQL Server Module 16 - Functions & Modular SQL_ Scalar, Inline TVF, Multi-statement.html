üîß Function Types Overview

Scalar vs Table-Valued Functions

Scalar Functions
Return single value: One scalar value per invocation
Usage: Calculations, formatting, data transformation
Performance: Row-by-row execution, can hurt performance
Limitations: Cannot use in WHERE clauses for index seeks
Best for: Simple calculations, reusable logic
Table-Valued Functions
Return table: Result set that can be queried
Inline vs Multi-statement: Inline expands to query, multi-statement materializes
Parameters: Can accept parameters for filtering
APPLY usage: Powerful with CROSS/OUTER APPLY
Best for: Modular queries, parameterized views
-- Scalar function example CREATE FUNCTION dbo.GetFullName(@FirstName VARCHAR(50), @LastName VARCHAR(50)) RETURNS VARCHAR(101) AS BEGIN RETURN RTRIM(@FirstName) + ' ' + RTRIM(@LastName); END; -- Usage (can hurt performance in large result sets) SELECT CustomerID, dbo.GetFullName(FirstName, LastName) AS FullName FROM Customers; -- Inline TVF example CREATE FUNCTION dbo.GetActiveOrders(@CustomerID INT) RETURNS TABLE AS RETURN ( SELECT OrderID, OrderDate, TotalAmount FROM Orders WHERE CustomerID = @CustomerID AND Status = 'Active' ); -- Multi-statement TVF example CREATE FUNCTION dbo.GetCustomerSummary(@StartDate DATE, @EndDate DATE) RETURNS @Summary TABLE ( CustomerID INT, OrderCount INT, TotalAmount DECIMAL(10,2) ) AS BEGIN INSERT INTO @Summary SELECT CustomerID, COUNT(*), SUM(TotalAmount) FROM Orders WHERE OrderDate BETWEEN @StartDate AND @EndDate GROUP BY CustomerID; RETURN; END;

‚ö° Why Inline TVFs Are Often Better

Performance Advantages of Inline TVFs

Inline TVF Benefits
Query expansion: Function code inlines into calling query, allowing optimization
Statistics awareness: Optimizer sees table structures and can use indexes
Parallel execution: Can participate in parallel query plans
Parameter sniffing: Benefits from parameter values for plan optimization
No materialization: Results stream directly without temp storage
Better performance: Often faster than multi-statement TVFs or views
Multi-statement TVF Limitations
Materialization: Results stored in temp table, then queried
Fixed estimates: Optimizer assumes 1 or 100 rows, poor cardinality
No statistics: Cannot use table statistics for optimization
Serial execution: Cannot run in parallel
Tempdb usage: Consumes tempdb space for result storage
Performance penalty: Extra I/O for materialization and re-querying
-- Inline TVF: Expands to efficient query SELECT c.CustomerName, o.OrderID, o.OrderDate FROM Customers c CROSS APPLY dbo.GetActiveOrders(c.CustomerID) o; -- Expands to (approximately): SELECT c.CustomerName, o.OrderID, o.OrderDate FROM Customers c INNER JOIN Orders o ON c.CustomerID = o.CustomerID WHERE o.Status = 'Active'; -- Multi-statement TVF: Materializes first, then joins SELECT c.CustomerName, s.OrderCount, s.TotalAmount FROM Customers c CROSS APPLY dbo.GetCustomerSummary('2023-01-01', '2023-12-31') s; -- Behind the scenes: Materialize temp table, then join -- Poor cardinality estimates lead to bad query plans

üîó APPLY + TVF Patterns

Advanced Query Patterns with Table-Valued Functions

CROSS APPLY
Inner join semantics: Only returns rows where TVF produces results
One-to-many: Each left row can produce multiple right rows
Parameterized correlation: TVF can use values from left table
Performance: Inline TVFs optimize well with CROSS APPLY
Use cases: Detail records, running totals, custom aggregations
OUTER APPLY
Left join semantics: Returns all left rows, NULLs for no TVF results
Optional relationship: TVF results are optional
Default values: Can provide defaults when TVF returns nothing
Performance: Same optimization benefits as CROSS APPLY
Use cases: Latest records, optional details, sparse relationships
-- CROSS APPLY: Get order details for each customer SELECT c.CustomerName, od.ProductID, od.Quantity FROM Customers c CROSS APPLY dbo.GetCustomerOrderDetails(c.CustomerID) od ORDER BY c.CustomerName, od.OrderID; -- OUTER APPLY: Get latest order for each customer (or NULL) SELECT c.CustomerName, ISNULL(lo.OrderDate, 'No orders') AS LastOrderDate, ISNULL(CAST(lo.TotalAmount AS VARCHAR), 'N/A') AS LastOrderAmount FROM Customers c OUTER APPLY ( SELECT TOP 1 OrderDate, TotalAmount FROM Orders WHERE CustomerID = c.CustomerID ORDER BY OrderDate DESC ) lo; -- Advanced pattern: Running totals with APPLY CREATE FUNCTION dbo.GetRunningTotal(@CustomerID INT, @UpToDate DATE) RETURNS TABLE AS RETURN ( SELECT SUM(TotalAmount) AS RunningTotal FROM Orders WHERE CustomerID = @CustomerID AND OrderDate <= @UpToDate ); SELECT o.OrderID, o.OrderDate, o.TotalAmount, rt.RunningTotal FROM Orders o CROSS APPLY dbo.GetRunningTotal(o.CustomerID, o.OrderDate) rt WHERE o.CustomerID = 123 ORDER BY o.OrderDate; -- APPLY with TOP for "latest N" pattern SELECT p.ProductName, recent.ReviewDate, recent.Rating FROM Products p OUTER APPLY ( SELECT TOP 3 ReviewDate, Rating FROM ProductReviews WHERE ProductID = p.ProductID ORDER BY ReviewDate DESC ) recent;
üí° Function Selection Guidelines: Prefer inline TVFs over multi-statement TVFs for better performance. Use scalar functions sparingly in WHERE clauses. Leverage APPLY for correlated subqueries and complex relationships.
‚ö†Ô∏è Performance Pitfalls: Scalar functions in SELECT lists can force row-by-row execution. Multi-statement TVFs can cause spool operations. Always test function performance with realistic data volumes.
