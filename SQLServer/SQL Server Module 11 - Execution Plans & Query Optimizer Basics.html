üîÑ Logical vs Physical Operators

Understanding Execution Plan Components

Logical Operators
What to do: Describe the operation conceptually (JOIN, SELECT, WHERE)
Algebraic representation: Mathematical description of query logic
Query structure: How the query is parsed and represented
Optimization input: Used by optimizer to find efficient physical plans
Independent of implementation: Same logical operators across database systems
Physical Operators
How to do it: Specific algorithms and data access methods
Implementation details: Index seeks, scans, sorts, joins
Cost-based: Each operator has I/O and CPU cost estimates
Execution path: Actual steps SQL Server takes to execute query
Performance metrics: Actual vs estimated rows, execution time
-- View execution plan (Ctrl+M in SSMS) SELECT c.CustomerName, o.OrderDate, o.TotalAmount FROM Customers c INNER JOIN Orders o ON c.CustomerID = o.CustomerID WHERE o.OrderDate >= '2023-01-01' ORDER BY o.TotalAmount DESC; -- Common physical operators you'll see: -- Index Seek: Efficient index lookup -- Index Scan: Full index traversal -- Table Scan: Full table read -- Nested Loops: Row-by-row join -- Merge Join: Sorted input join -- Hash Match: Hash-based join -- Sort: In-memory or tempdb sorting -- Stream Aggregate: Grouped aggregation

üîó Join Algorithms

Nested Loops, Merge Join, Hash Match

Algorithm Best For When Used Memory Usage Performance Characteristics
Nested Loops Small outer input, indexed inner One table much smaller, good index on join key Low Fast startup, scales poorly with large datasets
Merge Join Sorted inputs, large datasets Both inputs sorted on join key, equality joins Low Predictable performance, requires sorted data
Hash Match Unsorted large datasets No useful indexes, complex joins, large result sets High Good for large unsorted data, memory intensive
-- Nested Loops example (small outer, indexed inner) SELECT c.CustomerName, o.OrderID FROM ( SELECT TOP 10 CustomerID FROM Customers ORDER BY CustomerName ) c INNER JOIN Orders o ON c.CustomerID = o.CustomerID; -- Merge Join example (sorted inputs) SELECT c.CustomerName, o.OrderDate FROM Customers c INNER JOIN Orders o ON c.CustomerID = o.CustomerID ORDER BY c.CustomerID; -- Both tables sorted by CustomerID -- Hash Match example (large unsorted datasets) SELECT p.ProductName, s.QuantitySold FROM Products p INNER JOIN Sales s ON p.ProductID = s.ProductID WHERE s.SaleDate >= '2023-01-01'; -- Force specific join hint (use sparingly) SELECT * FROM Table1 t1 INNER HASH JOIN Table2 t2 ON t1.ID = t2.ID;

üëÉ Parameter Sniffing

Detection and Handling Strategies

Parameter Sniffing Explained
What it is: Optimizer uses parameter values from first execution to create plan
Problem: Cached plan may be suboptimal for different parameter values
Detection: Look for plans with high variance in actual vs estimated rows
Impact: Queries run fast for some parameters, slow for others
Common in: Stored procedures, prepared statements, forced parameterization
-- Example stored procedure with parameter sniffing issue CREATE PROCEDURE GetOrdersByCustomer @CustomerID INT AS BEGIN SELECT * FROM Orders WHERE CustomerID = @CustomerID ORDER BY OrderDate DESC; END -- First execution with small customer (creates nested loops plan) EXEC GetOrdersByCustomer @CustomerID = 1; -- Small customer, fast -- Second execution with large customer (reuses nested loops, slow) EXEC GetOrdersByCustomer @CustomerID = 999; -- Large customer, slow -- Solutions: -- 1. RECOMPILE hint (recompiles each time) CREATE PROCEDURE GetOrdersByCustomer @CustomerID INT AS BEGIN SELECT * FROM Orders WHERE CustomerID = @CustomerID ORDER BY OrderDate DESC OPTION (RECOMPILE); END -- 2. OPTIMIZE FOR hint (optimize for specific value) SELECT * FROM Orders WHERE CustomerID = @CustomerID ORDER BY OrderDate DESC OPTION (OPTIMIZE FOR (@CustomerID = 500)); -- Average case -- 3. Local variables (prevents sniffing) CREATE PROCEDURE GetOrdersByCustomer @CustomerID INT AS BEGIN DECLARE @CustID INT = @CustomerID; SELECT * FROM Orders WHERE CustomerID = @CustID ORDER BY OrderDate DESC; END

üìä Plan Analysis Checklist

‚úÖ Execution Plan Analysis

  • Check for actual execution plan (not estimated)
  • Compare actual vs estimated row counts
  • Look for expensive operators (high cost %)
  • Identify scans vs seeks (table/index scans)
  • Check for join algorithm appropriateness
  • Look for sort/hash warnings (memory spills)
  • Verify index usage and missing index suggestions
  • Check for parameter sniffing indicators
  • Monitor execution time and resource usage
  • Understand logical to physical operator mapping

üß™ Practice Exercises

  1. Plan Analysis: Execute queries and analyze their execution plans. Identify the physical operators used and explain why each algorithm was chosen.
  2. Join Algorithms: Write queries that force different join algorithms using hints. Compare performance and resource usage for each approach.
  3. Parameter Sniffing: Create a stored procedure that exhibits parameter sniffing. Implement different solutions (RECOMPILE, OPTIMIZE FOR, local variables) and measure performance.
  4. Optimization: Identify poorly performing queries in your environment. Analyze their execution plans and implement appropriate fixes.
  5. Advanced Analysis: Use DMVs like sys.dm_exec_query_stats and sys.dm_exec_cached_plans to analyze plan cache and identify optimization opportunities.
