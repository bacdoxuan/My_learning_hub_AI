üìä Database Normalization Overview
Normalization is the process of organizing data to reduce redundancy and improve data integrity by decomposing tables into smaller, related tables.
Normal Form Rule Problem Solved
1NF Atomic values, no repeating groups Multi-valued attributes
2NF 1NF + full dependency on PK Partial dependencies
3NF 2NF + no transitive dependencies Non-key dependencies
1Ô∏è‚É£ First Normal Form (1NF)
A table is in 1NF when all values are atomic, each row is unique, and there are no repeating groups.
‚ùå Violates 1NF - Bad Design
-- Problem: Phone numbers stored as comma-separated values CREATE TABLE customers_bad ( customer_id INT PRIMARY KEY, customer_name VARCHAR(100), phone_numbers VARCHAR(255) -- Contains '555-1234, 555-5678' );
‚úÖ 1NF Compliant - Good Design
-- Solution: Separate table for phone numbers CREATE TABLE customers ( customer_id INT PRIMARY KEY, customer_name VARCHAR(100) NOT NULL ); CREATE TABLE customer_phones ( phone_id INT PRIMARY KEY AUTO_INCREMENT, customer_id INT, phone_number VARCHAR(20) NOT NULL, phone_type VARCHAR(20), -- 'mobile', 'home', 'work' FOREIGN KEY (customer_id) REFERENCES customers(customer_id) );
‚úÖ Benefits: Each phone number is atomic, can be queried individually, and can have metadata (type)
2Ô∏è‚É£ Second Normal Form (2NF)
A table is in 2NF when it's in 1NF AND all non-key attributes are fully dependent on the entire primary key (no partial dependencies).
‚ùå Violates 2NF - Bad Design
-- Problem: Supplier info depends on product, not order CREATE TABLE order_items_bad ( order_id INT, product_id INT, supplier_name VARCHAR(100), supplier_address VARCHAR(255), quantity INT, PRIMARY KEY (order_id, product_id) ); -- Issue: supplier_name and supplier_address depend only on product_id
‚úÖ 2NF Compliant - Good Design
CREATE TABLE orders ( order_id INT PRIMARY KEY AUTO_INCREMENT, customer_id INT, order_date DATE, FOREIGN KEY (customer_id) REFERENCES customers(customer_id) ); CREATE TABLE order_items ( order_item_id INT PRIMARY KEY AUTO_INCREMENT, order_id INT, product_id INT, quantity INT, unit_price DECIMAL(10, 2), FOREIGN KEY (order_id) REFERENCES orders(order_id), FOREIGN KEY (product_id) REFERENCES products(product_id) ); CREATE TABLE products ( product_id INT PRIMARY KEY, product_name VARCHAR(100), supplier_id INT, FOREIGN KEY (supplier_id) REFERENCES suppliers(supplier_id) ); CREATE TABLE suppliers ( supplier_id INT PRIMARY KEY, supplier_name VARCHAR(100), supplier_address VARCHAR(255) );
3Ô∏è‚É£ Third Normal Form (3NF)
A table is in 3NF when it's in 2NF AND there are no transitive dependencies (non-key attributes don't depend on other non-key attributes).
‚ùå Violates 3NF - Bad Design
-- Problem: Course instructor depends on course, not student directly CREATE TABLE enrollments_bad ( student_id INT, course_id INT, course_name VARCHAR(100), instructor_name VARCHAR(100), instructor_office VARCHAR(50), PRIMARY KEY (student_id, course_id) ); -- Issue: instructor_name and instructor_office depend on course_id only
‚úÖ 3NF Compliant - Good Design
CREATE TABLE students ( student_id INT PRIMARY KEY, student_name VARCHAR(100), email VARCHAR(100) ); CREATE TABLE courses ( course_id INT PRIMARY KEY, course_name VARCHAR(100), instructor_id INT, FOREIGN KEY (instructor_id) REFERENCES instructors(instructor_id) ); CREATE TABLE instructors ( instructor_id INT PRIMARY KEY, instructor_name VARCHAR(100), instructor_office VARCHAR(50), department_id INT ); CREATE TABLE enrollments ( enrollment_id INT PRIMARY KEY AUTO_INCREMENT, student_id INT, course_id INT, enrollment_date DATE, FOREIGN KEY (student_id) REFERENCES students(student_id), FOREIGN KEY (course_id) REFERENCES courses(course_id) );
üí° Benefits of 3NF: Eliminates redundant instructor data, prevents update anomalies when instructor changes office
üîó Relationship Types
Relationship Description Example
One-to-One (1:1) Each record in A links to exactly one in B Employee ‚Üî Office
One-to-Many (1:N) One record in A links to many in B Department ‚Üî Employees
Many-to-Many (M:N) Many records in A link to many in B Students ‚Üî Courses
1:1 Relationship Example
CREATE TABLE employees ( employee_id INT PRIMARY KEY, employee_name VARCHAR(100), office_id INT UNIQUE -- Each employee has one office ); CREATE TABLE offices ( office_id INT PRIMARY KEY, office_location VARCHAR(100) ); ALTER TABLE employees ADD FOREIGN KEY (office_id) REFERENCES offices(office_id);
üë• One-to-Many (1:N) Relationship
CREATE TABLE departments ( department_id INT PRIMARY KEY, department_name VARCHAR(100) ); CREATE TABLE employees ( employee_id INT PRIMARY KEY, employee_name VARCHAR(100), department_id INT, FOREIGN KEY (department_id) REFERENCES departments(department_id) );
‚úÖ One department can have many employees. Foreign key goes on the "many" side.
üéì Many-to-Many (M:N) Relationship
CREATE TABLE students ( student_id INT PRIMARY KEY, student_name VARCHAR(100) ); CREATE TABLE courses ( course_id INT PRIMARY KEY, course_name VARCHAR(100) ); -- Junction table to resolve M:N relationship CREATE TABLE student_courses ( student_course_id INT PRIMARY KEY AUTO_INCREMENT, student_id INT, course_id INT, enrollment_date DATE, grade VARCHAR(2), UNIQUE(student_id, course_id), FOREIGN KEY (student_id) REFERENCES students(student_id), FOREIGN KEY (course_id) REFERENCES courses(course_id) );
üí° Junction table (bridge table) contains foreign keys to both entities, resolving the many-to-many relationship into two one-to-many relationships.
üîë Primary Keys
-- Single column primary key CREATE TABLE categories ( category_id INT PRIMARY KEY AUTO_INCREMENT, category_name VARCHAR(100) NOT NULL ); -- Composite primary key CREATE TABLE inventory_locations ( warehouse_id INT NOT NULL, product_id INT NOT NULL, quantity INT DEFAULT 0, PRIMARY KEY (warehouse_id, product_id) ); -- Natural primary key (business logic) CREATE TABLE user_accounts ( username VARCHAR(50) PRIMARY KEY, email VARCHAR(100) NOT NULL, created_date DATE );
üîó Foreign Keys
-- Basic foreign key CREATE TABLE orders ( order_id INT PRIMARY KEY AUTO_INCREMENT, customer_id INT NOT NULL, order_date DATE, FOREIGN KEY (customer_id) REFERENCES customers(customer_id) ); -- Foreign key with referential integrity actions CREATE TABLE order_items ( order_item_id INT PRIMARY KEY AUTO_INCREMENT, order_id INT NOT NULL, product_id INT NOT NULL, quantity INT, FOREIGN KEY (order_id) REFERENCES orders(order_id) ON DELETE CASCADE -- Delete items when order is deleted ON UPDATE CASCADE, FOREIGN KEY (product_id) REFERENCES products(product_id) ON DELETE RESTRICT -- Prevent product deletion if items exist );
‚úÖ ON DELETE CASCADE: Automatically delete related records
‚úÖ ON DELETE RESTRICT: Prevent deletion if related records exist
‚úÖ ON DELETE SET NULL: Set foreign key to NULL
‚õî Other Constraints

Constraint Purpose
NOT NULL Column must have a value
UNIQUE All values must be different
CHECK Values must meet condition
DEFAULT Default value if none provided

CREATE TABLE customers ( customer_id INT PRIMARY KEY AUTO_INCREMENT, email VARCHAR(100) UNIQUE NOT NULL, password_hash VARCHAR(255), created_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP ); CREATE TABLE addresses ( address_id INT PRIMARY KEY AUTO_INCREMENT, customer_id INT NOT NULL, street VARCHAR(100), city VARCHAR(50), state VARCHAR(50), postal_code VARCHAR(20), address_type ENUM('shipping', 'billing'), FOREIGN KEY (customer_id) REFERENCES customers(customer_id) ON DELETE CASCADE ); CREATE TABLE categories ( category_id INT PRIMARY KEY AUTO_INCREMENT, category_name VARCHAR(100) UNIQUE NOT NULL ); CREATE TABLE products ( product_id INT PRIMARY KEY AUTO_INCREMENT, category_id INT NOT NULL, product_name VARCHAR(100) NOT NULL, description TEXT, price DECIMAL(10, 2) NOT NULL CHECK (price > 0), quantity_in_stock INT DEFAULT 0, created_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP, FOREIGN KEY (category_id) REFERENCES categories(category_id) );
